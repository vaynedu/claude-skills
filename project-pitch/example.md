 Round 1 · 范文答案                                                                                                         
                                                                                                                             
  （钩子——讲产品，让面试官有画面）                                                                                           
                                                                                                                             
  "xxxxx 是 pap 游戏中台下的营销服务，xxxx这类出海游戏，玩家在游戏商城里看到的每个商品、每个折扣标签、每个'限购 3
   件'的提示，以及点击充值时弹出的收银台支付方式，背后都是这个服务在下发。                                                   
                                                                                                                             
  （业务价值——让技术外的人也能听懂）                                                                                         
  
  它解决的是游戏商品运营链路上的三类问题：上架下发（什么商品在什么时间、对什么玩家、用什么价格展示）、促销与限购（首充双倍、 
  月卡周卡、BP 通行证、回流礼包这类复杂规则）、发货前校验（下单到发货这一跳的幂等、限购扣减、互斥判断）。对运营来说，它是个「
  配置即生效」的商品管理后台；对玩家来说，它决定了商城页打开后看到什么。                                                     
                  
  （系统位置——边界清晰）                                                                                                     
  
  在链路上它是个典型的中台服务。对上承接三类调用方：游戏客户端、H5 充值中心、游戏 SDK，通过一份 proto 暴露 HTTP 和 gRPC      
  双协议；对下两条线——一条同步调用，拉取xxxxxx 拿元商品、gm 拿角色等级/回流时间/封号状态、area 拿地区币种；一条异步，消费
   tMQ 的 order 订单流水做后处理。                                                                               
                  
  （数字撑规模感）                                                                                                           
  
  规模上：1 个 proto、13 个 RPC、8 张核心业务表、9 个 service 子包、44 个业务过滤码、6 个下游 HTTP client，支持多租户 +      
  多区服 + 多平台的六元组数据隔离，出海覆盖 PSN、Epic、Apple 兑换码，以及 7 种以上海外支付通道（antom/adyen/mercadopago/dana
  等）。                                                                                                                     
                  
  （技术亮点——「是什么 + 为什么 + 取舍」）                                                                                   
  
  技术上重点讲两个亮点：                                                                                                     
                  
  第一个是配置数据驱动 + 三层缓存一致性。8 张运营配置表全部做了内存化：服务启动时 LoadCache 预热到 sync.Map，运行时 etcd     
  watch 订阅 13 个 resource entry，B 端运营一保存，watcher 回调触发 DAO 重新拉 DB 刷缓存；为了防 etcd
  消息丢失，互斥关系这类二阶衍生数据还挂了一个 5 分钟定时兜底重建。缓存读出来时每次 Clone                                    
  一份返回，避免外层污染。这个设计的价值是运营改活动不发版、秒级生效；取舍是每个租户要提前进白名单
  InMemoryClientId，不是开放式多租户。

  第二个是发货校验的 Lua 原子脚本。发货这一跳是最敏感的，一次调用要同时解决 5                                                
  件事——订单幂等、商品限购扣减、互斥商品校验、前置依赖校验、首充标记
