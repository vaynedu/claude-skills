# 项目讲述技能 · 30 秒钩子 + 3 分钟全貌

把任何技术项目讲清楚的通用模板。**用于面试 / 跨团队分享 / 向上汇报 / 新人 onboarding / 给产品运营介绍**。

---

## 触发条件

用户需要向某人讲述一个项目/服务时。典型表述：

- "帮我讲讲 XX 项目"
- "怎么给别人介绍这个服务"
- "面试怎么讲这段经历"
- "我要做一个技术分享，怎么组织"
- "给新同事 onboarding 怎么讲"
- "给领导汇报 XX 项目进展"

**不适用**：写技术文档、写设计方案、写 README（那些是落地成文字，本 skill 只管口头/短时讲述）。

---

## 核心心法

**按"听众脑子里缺什么"组织，不是按代码结构组织。**

听众脑子里会依次问 5 个问题：

1. 这是**什么**？（业务）
2. 给**谁**用？解决**什么问题**？（价值）
3. 它在**哪**？上下游是谁？（边界）
4. **多大**？（规模）
5. 技术上**难在哪 / 怎么解**？（亮点）

讲述顺序固定：**业务 → 价值 → 边界 → 规模 → 技术**。

---

## 5 段式黄金结构（总时长 3 分钟）

| 段 | 时长 | 内容 | 作用 | 关键句式 |
|---|---|---|---|---|
| 1 · 钩子 | 30s | 一句话说清产品场景 | 给画面 | "XX 是 YY 的 ZZ——用户在 ZZ 里做的每一件事背后都是它" |
| 2 · 业务价值 | 30s | 它在业务里承担什么 | 证明你懂业务 | "承担三件事：A、B、C" |
| 3 · 上下游边界 | 45s | 对上几类调用方 / 对下几条链路 | 画系统边界 | "对上 X 类调用方...对下两条线..." |
| 4 · 规模 | 15s | 数字撑场 | 让抽象落地 | "X 张核心表 / Y 个接口 / Z 个活动" |
| 5 · 技术亮点 | 60s | 1-2 个亮点 | 证明设计思考 | "是什么 + 为什么 + 取舍" |

**总和**：~3 分钟，给听众留出追问空间。

## 填空式模板（让用户填，或你帮他填）

```
【段 1 · 钩子】
{项目名} 是 {产品线/公司} 的 {产品形态}——用户在 {具体场景} 里做的每一件事：
{功能1}、{功能2}、{功能3}，背后都是这个服务在处理。

【段 2 · 业务价值】
从业务目标看，它承担 {领域} 里的 {目标1}、{目标2}、{目标3} 三件事：
{目标1} 是 {功能/场景}；
{目标2} 是 {功能/场景}；
{目标3} 是 {功能/场景}。

【段 3 · 上下游边界】
它在链路上是个 {中间层/前置层/后端}。
对上承接 {N} 类调用方：{类型1}、{类型2}、{类型3}，通过 {协议} 暴露 {M} 个接口。
对下 {N} 条线：
一条是 {下游1}，负责 {做什么}；
另一条是 {下游2}，负责 {做什么}。

【段 4 · 规模】
规模上，{核心表数}、{接口数}、{数据量/QPS/用户量}。

【段 5 · 技术亮点】
技术上我想重点讲 {N} 个亮点：

第一个是 {亮点名}。
  · 是什么：{核心抽象 / 模式 / 设计}
  · 为什么：{解决什么问题 / 原设计有什么痛点}
  · 取舍：{这个设计不完美的地方，以及为什么可以接受}

第二个是 {亮点名}。
  · {结构同上}
```

## 3 个黄金锚点（让讲述有记忆点）

好的项目讲述会让听众记住 3 个关键词。讲完后听众脑子里应该留下：

1. **1 个业务场景**（不是技术词）
2. **1 个数字**（规模感）
3. **1 个技术抽象**（亮点）

别人听完第二天能复述出这 3 个词，就是成功。

**例子**（kf-wx 版）：

- 业务锚点："玩家在小程序里领月度礼"
- 数字锚点："9 张核心表" 或 "20+ 活动策略"
- 技术锚点："ActivityProcessor 模式"

---

## 6 条自检清单（讲完后必做）

- [ ] 第一句话**有画面感**吗？非技术的人能想象出使用场景吗？
- [ ] 有**至少 3 个数字**吗？（表数、接口数、QPS、活动数任选）
- [ ] 每个技术亮点都讲了"**为什么这么设计**"，不是只讲"是什么"？
- [ ] 有主动说"**取舍 / 不完美 / 妥协**"吗？（缺这个会显得不可信）
- [ ] **3 分钟内**能讲完吗？（超 5 分钟一定要砍）
- [ ] 讲完后听众能**追问出至少 2 个问题**吗？（只能问出 1 个说明钩子不够）

---

## 5 大反模式（千万别踩）

| 反模式 | 症状 | 正确做法 |
|---|---|---|
| ❌ **念目录** | "我们有 control、service、repo 层..." | 讲**业务能力**，不是代码结构 |
| ❌ **堆技术栈** | "Go + gRPC + Gin + GORM + Kafka + Redis + ..." | 技术栈放到"规模"那 15 秒里带一下，不作为亮点 |
| ❌ **完美叙述** | "我们的系统稳定高效..." | 主动暴露取舍，更可信 |
| ❌ **只讲是什么** | "我们用了 xx 设计模式" | 永远补"**解决了什么问题**" |
| ❌ **讲到哪算哪** | 开放式结尾，听众不知道该问啥 | 留 1-2 个**明显的追问钩子** |

---

## 按听众调整（6 个场景）

| 场景 | 是否适用 | 调整方法 |
|---|---|---|
| 技术面试自我介绍 | ✅ 完全适用 | **加强**取舍和踩坑（第 5 段留 1 个亮点即可） |
| 跨团队分享 | ✅ 适用 | **加强**业务价值段，淡化实现细节 |
| 向上汇报 | ✅ 适用 | **加强**规模和产出数据，第 5 段换成"下季度要做什么" |
| 新人 onboarding | ✅ 适用 | **加强**上下游段，技术亮点后给**源码入口** |
| 给产品 / 运营讲 | ⚠️ 要改 | **砍掉**技术亮点段，换成"你能怎么用它"（能力边界 + 配置方式） |
| 给客户讲（ToB） | ⚠️ 要改 | **砍掉**技术细节，强调**业务价值和 ROI** |

## 执行流程（当用户触发此技能时）

### Step 1 · 理解上下文

问用户（如果没给）：

1. **要讲什么项目 / 系统？** 如果在仓库里，可以让用户指路径或你自己探索
2. **讲给谁？**（面试官 / 同事 / 领导 / 新人 / 产品 / 客户）
3. **多长时间？**（默认 3 分钟，可以 30 秒压缩 或 15 分钟展开）
4. **特别想突出什么？**（是否有用户自己觉得的亮点）

### Step 2 · 探索项目（必要时）

如果用户说"你直接帮我讲"，而你不熟这个项目：

- 读 README / CLAUDE.md / docs/
- `git log --oneline | head -30` 看最近迭代
- `ls` 看目录结构判断分层
- 重点找：**业务入口文件**（main.go / app.go / control 层）、**核心数据模型**（model / schema）、**外部依赖**（config 配置、Kafka topic、下游服务）

### Step 3 · 按模板填充 5 段

严格按业务 → 价值 → 边界 → 规模 → 技术的顺序。

**关键动作**：

- 第 1 段务必找一个**具体的用户场景**作为钩子（不是"这是一个 XX 系统"）
- 第 4 段至少给 3 个数字，如果实在没有，用"X 张核心表 / Y 个接口"凑
- 第 5 段亮点最多 2 个（1 个也行），每个必须讲"是什么 + 为什么 + 取舍"

### Step 4 · 按听众调整

根据 Step 1 的听众类型，参考上面"按听众调整"表做增删：

- 面试：砍第 4 段，扩第 5 段
- 领导：扩第 4 段，第 5 段换成规划
- 新人：第 5 段后加"推荐看这几个文件"
- 产品：砍第 5 段技术亮点，换能力边界

### Step 5 · 给出完整范文 + 结构说明表

范文要**逐字可读**（让用户能直接照着讲），同时给一张"为什么这样讲"的结构说明表（教用户模板的内化逻辑）。

### Step 6 · 自检清单

讲完后，主动用上面 6 条自检清单过一遍，指出可能的弱点让用户知道：

- "这版里数字只有 2 个，如果能拿到 QPS 会更好"
- "第 5 段只讲了 1 个亮点，面试时可以追加 1 个"
- "结尾有点硬，加一句'如果深入讲还能展开 XX'会更好"

## 反面示例 vs 正面示例

### ❌ 反面（常见 bad case）

> "这是一个 Go 微服务，我们用了分层架构，包括 control、service、repo、model 层，技术栈是 Go + Gin + GORM + Redis + Kafka。我负责了用户签到模块的开发，用了 GORM 做数据库访问，Redis 做缓存。我们还集成了 SchedulerX 做定时任务。"

**问题**：

- 第一句没画面，面试官听完还是不知道"到底是个啥"
- 全是技术栈堆砌
- 没讲业务价值、没数字、没设计思考
- 结尾开放式，面试官不知道追问啥

### ✅ 正面（对比）

> "kf-wx 是叠纸微信小程序的活动运营后台——玩家在暖暖游戏的小程序里做的每一件事：绑定账号、签到、领 VIP 月度礼，背后都是这个服务在处理。它承担私域运营里的拉新、转化、召回三件事。
>
> 在链路上是个中间层，对上承接小程序前端、H5、游戏 SDK 三类调用方，一共 30 多个接口。对下两条线：push-center 发奖、消费 BI 的 VIP 充值 Kafka 更新人群包。
>
> 规模上是 9 张核心表 + 20 多个活动策略。
>
> 技术上最想讲的是 ActivityProcessor 模式：不同活动（签到 / 月度礼 / 生日礼）的校验规则差异很大，抽了一个 Processor 接口，4 个实现各自定义 Validate / ResolveUserSegments / PostProcess，主发奖链路一行没改过。取舍是工厂 switch 还没做注册化，考虑到活动类型增长慢，这个妥协是合理的。"

**为什么好**：

- 第一句"玩家在小程序领月度礼" → 画面
- 拉新 / 转化 / 召回 → 业务三件事
- 数字：30 / 9 / 20 → 规模感
- ActivityProcessor + 取舍 → 设计思考
- 3 分钟讲完，结尾能追问出 3 个问题

## 我应该做什么 / 不做什么

**应该**：

- 帮用户按模板填空，产出**逐字可读**的范文
- 主动探索项目上下文（如果在 repo 里）
- 给出"结构说明表"教用户内化逻辑
- 用自检清单指出弱点

**不应该**：

- 越俎代庖写一份长篇文档（那是另一回事，用 system-design-doc 或写 README）
- 堆技术栈（这个 skill 反对的就是这个）
- 写成行业/架构分析报告（那是咨询，不是项目讲述）

## 参考成功案例

**kf-wx 项目讲述**（2026-04-27 实战）：

- 用户场景：面试模式 Round 1 自我介绍
- 讲述时长：3 分钟
- 覆盖要素：业务 / 价值 / 边界 / 规模 / 2 个技术亮点
- 追问触发：面试官成功追问到 Processor 模式和幂等三层
- 内化后：用户能自主复述，不再依赖讲稿
