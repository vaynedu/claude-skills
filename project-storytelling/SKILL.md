# 项目讲述技能

帮你把技术项目整理成**有说服力的叙事**，用于面试、述职、技术分享、周报汇总。

核心原则：**讲决策，不讲罗列。** 每个技术选择背后对应一个业务问题，把问题和选择配对讲出来。

## 触发条件

- "帮我讲讲这个项目"
- "怎么介绍我做的东西"
- "面试怎么讲这个项目"
- "帮我总结项目亮点"
- "述职/周报/技术分享素材"

## 工作流

### Phase 1：扫描项目，提取素材

```
步骤：
1. 读取 CLAUDE.md / README — 项目定位、技术栈
2. 读取 proto/*.proto — 接口定义，理解业务能力边界
3. 读取 control/ + service/ — 核心业务逻辑
4. 读取 common/errors.go — 错误处理策略
5. git log --oneline -20 — 近期演进方向
6. 扫描 utils/notify、config、database — 基础设施接入情况
```

从代码中提取：
- **业务场景**：这个服务解决什么业务问题
- **架构决策**：分层、协议、存储选型及其 why
- **防御性设计**：锁、去重、校验、限流、降级
- **可观测性**：日志、告警、指标、链路追踪
- **已知 trade-off**：有意识接受的缺陷

### Phase 2：生成叙事框架

按以下结构组织，每个项目都套这个模板：

```
## 一句话定位（电梯演讲）
{服务名} 是 {谁} 用的 {做什么} 的服务，核心解决 {什么问题}，日均处理量 {X}。

## 业务场景（30 秒讲清楚）
用户是谁 → 他们要做什么 → 不做这个服务会怎样

## 架构决策（重点：WHY）
| 我做了什么 | 为什么这么做（业务问题） | 不这么做会怎样 |

## 值得深聊的细节（2-3 个亮点）
每个亮点结构：
- 问题：遇到了什么
- 方案：怎么解决的
- 取舍：放弃了什么、接受了什么代价
- 效果：解决后的数据/结果（如果有）

## 知道自己缺什么（展现成熟度）
主动说出 1-2 个待改进点 + 为什么还没做 + 下一步计划

## 数据支撑（如果有）
QPS、延迟、成功率、处理量等量化指标
```

### Phase 3：输出格式

根据用户场景选择输出格式：

| 场景 | 输出 |
|------|------|
| 面试准备 | 口语化叙事 + 追问预判（面试官可能追问什么） |
| 述职/周报 | 结构化要点 + 量化数据 |
| 技术分享 | 问题驱动的演讲大纲 |
| 快速总结 | 一页纸项目概览 |

## 叙事原则

### 必须遵循

1. **问题先行**：先讲业务痛点，再讲技术方案。"运营一次导入 500 条，不能重复发" 比 "用了 Redis SETNX" 有力
2. **决策配对**：每个技术选择必须配一个业务原因。没有 why 的技术点不值得讲
3. **承认缺陷**：主动暴露 1-2 个已知不足 + 改进计划，展现全局思维
4. **量化优先**：能用数字的不用形容词。"P99 < 200ms" 比 "性能很好" 有效
5. **取舍显功力**：讲清楚你放弃了什么、为什么可以接受，比讲"用了什么"更有区分度

### 必须避免

| 反模式 | 为什么差 | 应该怎么讲 |
|--------|---------|-----------|
| 罗列技术栈 | "Go + MySQL + Redis + K8s" 谁都会列 | 讲每个选型解决了什么问题 |
| 只讲做了什么 | "用了分布式锁" 没有信息量 | 讲为什么需要锁、锁的粒度怎么选的 |
| 回避缺陷 | 显得没有反思 | 主动说出来，附带改进思路 |
| 过度技术细节 | 听众没有你的上下文 | 先讲场景，再讲方案，按需深入 |
| 贬低项目规模 | "就是个 CRUD" 自己先矮了 | 任何系统都有设计决策，讲决策不讲规模 |
| 堆砌流行词 | "云原生微服务" 没有实质 | 讲具体解决了什么分布式问题 |

## 追问预判模板

面试场景下，为每个亮点准备追问：

```
亮点：Redis 防重复提交
├── Q: 为什么不用 DB 唯一索引？
│   A: 4 字段联合索引代价大，且业务只需 1h 内去重
├── Q: Redis 挂了怎么办？
│   A: 当前跳过去重（有意识的降级），可接受短时间重复
├── Q: 1h TTL 够吗？
│   A: 业务上运营不会 1h 内重复操作，超过 1h 可能是有意的修改
└── Q: 并发安全吗？
    A: SETNX 原子操作，天然并发安全
```

## 高级：对比式叙事

如果做了改进或重构，用 Before/After 结构讲更有冲击力：

```
Before: 导出没有并发锁，同一客户端连续点 3 次导出按钮，3 个查询同时打 DB
After: per-client 导出锁，SETNX TTL 600s，重复请求直接返回"导出进行中"
效果: DB 查询并发从 N 降到 1，消除了重复导出的风险
```

## 输出要求

- 中文输出
- 口语化，不要书面语
- 避免技术术语堆砌，用人话解释
- 面试场景下控制在 3 分钟内能讲完核心内容
